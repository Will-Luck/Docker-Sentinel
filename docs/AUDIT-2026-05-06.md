# Docker-Sentinel Comprehensive Audit — 2026-05-06

**Triggered by:** n8n container stuck at v2.9.4 despite "auto" update policy (~11 minor versions behind upstream 2.20.2).

**Scope:** Full Tier B (static + repo + CVE + changelog) + initial Tier C (test cluster prep).

**Total findings:** 47 across 6 workstreams. **1 critical**, **11 high**, **10 medium**, **25 low**. **0 CVEs.**

## Executive summary

Sentinel's update logic works exactly as coded. The n8n bug is a **design–intent mismatch** between operator expectation ("auto means update across versions") and implementation (`auto` is gated by `SemverScope`, default `relaxed` = patch-only within `major.minor` for 3-part tags). Same root cause was independently surfaced from the UX side: the `container.html` policy descriptions promise behaviour the engine doesn't deliver. This pattern — UI label vs backend semantic mismatch — appears in at least three places (`auto`, `pinned`, `:latest auto-update toggle`).

The audit also surfaced **one critical security finding**: BoltDB stores Docker Hub PAT and Gotify token in cleartext at rest, both leaked into audit terminal output during this run. **Tokens are not publicly exposed** (not in any git repo, not in env files); local exposure is `.57` sudo + this session's transcripts + Anthropic API conversation logs. Rotation deferred to morning (manual UI required; threat model accepted).

The single largest concentration of risk is `internal/web/static/app.js` — a 5,200-line monolith carrying all top-15 cyclomatic-complexity hotspots (highest: `initSSE` cycl. 94). Currently mid-refactor: duplicate `initSSE` and `initSettingsPage` exist in both `app.js` and the new `src/js/` modules, drift risk between them.

## Root-cause finding (the n8n bug)

| Layer | Evidence |
|---|---|
| **Operator action** | n8n deployed via Portainer stack with `image: n8nio/n8n:2.9.4` (pinned, not `:latest`) |
| **Sentinel policy resolution** | `internal/engine/policy.go:31-46` `ResolvePolicy` — versioned tag → falls through to `defaultPolicy` (which user has set to `auto`) |
| **Sentinel version filter** | `internal/registry/tags.go:208 NewerVersionsScoped` — for 3-part tag `2.9.4` under `ScopeDefault` (relaxed), filter is `sv.Major == 2 && sv.Minor == 9` |
| **Result** | n8n's only available 2.9.x version IS 2.9.4 (project moved past 2.9 long ago to 2.20.2). Filter empties → "up to date" forever |

**Fix paths** (pick one):
- (a) **Per-container**: edit Portainer stack, change `image: n8nio/n8n:2.9.4` → `n8nio/n8n:latest`. Sentinel will then digest-compare and auto-update on every release. **Cleanest.**
- (b) **Per-container label**: add Docker label `sentinel.semver=major` (or `minor`) to the n8n compose entry. Allows cross-version updates without changing tag.
- (c) **Global**: bump Sentinel global `version_scope` from `default` → `major`. Fixes everything in one switch but **could push Postgres 16→17** next scan (data corruption risk).

Recommend (a) for n8n. Recommend (b) for other pinned-but-want-updates containers. Postgres should remain pinned regardless.
